# Shepherd 设计文档

## 问题定义

基于 320 个子代理任务的统计分析：

- **超时率**：19.4%（62/320）
- **超时占失败比例**：71.3%
- **编码任务完成率**：57.1%（最弱）
- **趋势**：4月 64% → 6月 83%（波动大）

**核心矛盾**：系统无法区分两种状态 ——
- A：子代理在等 LLM 响应 / 编译 / API 返回（活着但慢）
- B：子代理陷入死循环 / 卡死（真死了）

当前做法是一刀切 kill，A 类被误杀。

## 架构设计

### 混合架构

```
┌─────────────────────────────────────────────┐
│              本地层（确定性，毫秒级）            │
│                                               │
│  心跳检测 → 时间阈值 → 续期/kill              │
│  进度追踪 → 文件变更检测                       │
│  统计仪表盘 → 数据聚合                         │
│  超时判定 → 硬上限 + 活动超时                   │
└───────────────────────┬─────────────────────┘
                        │ 触发条件
                        ↓
┌─────────────────────────────────────────────┐
│              LLM 层（智能判断，按需调用）        │
│                                               │
│  ① 超时后诊断：分析子代理输出，判断死因          │
│  ② 重派策略：基于死因决定怎么拆任务/换方法       │
│  ③ 异常检测：心跳正常但进度为零 → LLM 审查      │
│  ④ 任务分类增强：关键词匹配不到时 LLM 兜底分类   │
│  ⑤ 周报分析：超时趋势 → 根因分析 → 改进建议     │
└─────────────────────────────────────────────┘
```

**核心原则**：本地管"什么时候该干预"，LLM 管"干预什么、怎么干预"。

### 模块设计

#### 1. 任务分类器（Classifier）

基于关键词匹配，将任务描述映射到预定义类型，每种类型有独立的超时策略。

| 类型 | 基础超时 | 最大续期 | 心跳间隔 | 最大续期次数 |
|------|---------|---------|---------|------------|
| coding | 600s | 1200s | 30s | 3 |
| external_api | 300s | 900s | 45s | 4 |
| file_ops | 300s | 600s | 30s | 3 |
| diagnostic | 180s | 360s | 30s | 2 |
| data_heavy | 600s | 1800s | 60s | 4 |

#### 2. 心跳引擎（Heartbeat）

基于文件系统的心跳协议：

- **子代理侧**：启动时创建心跳文件 → 每次工具调用前更新 → 完成时删除
- **主会话侧**：监控脚本每 30s 检查所有心跳文件

心跳文件路径：`/tmp/shepherd/heartbeat-{taskId}.json`

心跳判定逻辑：
- 心跳停止 > 60s → 预警
- 心跳停止 > 120s → 真超时 → 尝试续期或 kill
- 总时长 > 最大续期 → 强制终止

#### 3. 动态超时引擎（Timeout Engine）

状态机：
```
RUNNING → (心跳停止) → WARNING → (120s未恢复) → TIMEOUT → RESPAWN/ESCALATE
```

续期规则：
- 续期 1-2 次：静默
- 续期 3 次：通知用户
- 续期 4 次：建议拆分任务
- 续期 5 次：强制 kill，不再续期

#### 4. 进度追踪器（Progress Tracker）

记录每个步骤的执行状态，支持断点续传。

进度文件路径：`/tmp/shepherd/progress-{taskId}.json`

超时 kill 后重派时，自动注入断点信息到新 task 描述。

#### 5. 超时诊断器（Diagnoser，LLM 层）

超时发生后，生成诊断提示词，由 LLM 分析：
- 死因分析
- 是否值得重派
- 重派策略（拆分/换方法/简化）

#### 6. 统计仪表盘（Dashboard）

实时统计：
- 今日/7天/全量超时率
- 按任务类型分类统计
- 续期次数分布
- 平均运行时长

## 数据流

```
spawn 子代理
    ↓
classify.sh → 任务类型 + 超时配置
    ↓
heartbeat.sh start → 创建心跳文件
progress.sh init → 创建进度文件
    ↓
子代理执行中
    ↓ heartbeat.sh beat（每次工具调用）
    ↓ progress.sh step（每个步骤完成）
    ↓ progress.sh file（文件变更）
    ↓
monitor.sh（每 30s cron）
    ↓
    ├── 心跳正常 → 静默
    ├── 心跳停止 > 60s → WARNING
    ├── 心跳停止 > 120s → 续期 or KILL
    │     ↓ KILL
    │   diagnose.sh → LLM 诊断
    │     ↓
    │   决策：重派 or 上报
    │     ↓ 重派
    │   progress.sh export → 断点信息
    │     ↓
    │   spawn 新子代理（含断点信息）
    │
    └── 总时长 > 最大续期 → MAX_KILL
```

## 预期效果

| 指标 | 当前 | Phase 1 后 | Phase 2 后 | 最终目标 |
|------|------|-----------|-----------|---------|
| 超时率 | 19.4% | ~12% | ~6% | **< 5%** |
| 编码完成率 | 57.1% | ~70% | ~85% | **> 90%** |
| 误杀率 | 高 | 中 | 低 | **极低** |

## 未来扩展

1. **OpenClaw 插件化**：封装为原生插件，通过 hooks 自动注入心跳协议
2. **WebSocket 实时推送**：超时事件实时推送到飞书/仪表盘
3. **自适应阈值**：基于历史数据自动调整超时阈值
4. **跨 Agent 对比**：不同模型的超时率对比分析
